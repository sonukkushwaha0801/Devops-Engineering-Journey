# Methods to Pass Variables to Terraform

# Why Terraform Needs Variable Input Methods

Terraform configurations are designed to be reusable.

Instead of hardcoding values such as:

* Environment names
* Instance types
* Region names
* VPC IDs
* Database names

Terraform allows values to be supplied dynamically.

This enables:

* Multi-environment deployments
* Reusable modules
* CI/CD automation
* Infrastructure standardization

---

# Example Variable

```hcl
variable "instance_type" {
  description = "EC2 Instance Type"
  type        = string
}
```

Terraform must receive a value for this variable before creating infrastructure.

---

# Method 1: Interactive Input Prompt

## What is it?

If a variable:

* has no default value
* is not present in tfvars files
* is not passed through CLI
* is not passed through environment variables

Terraform prompts the user to enter the value.

---

## Example

### variables.tf

```hcl
variable "instance_type" {
  type = string
}
```

### Command

```bash
terraform plan
```

### Output

```text
var.instance_type
Enter a value:
```

User enters:

```text
t3.micro
```

Terraform then continues execution.

---

## Advantages

* Very simple
* Good for learning
* No additional configuration required

---

## Disadvantages

* Requires manual interaction
* Not suitable for automation
* Cannot be used in CI/CD pipelines
* Easy to make typing mistakes

---

## Production Usage

Rarely used.

Mostly useful for:

* Labs
* Practice environments
* Small personal projects

---

# Method 2: Command Line Variables (-var)

Terraform allows variables to be passed directly from the command line.

---

## Syntax

```bash
terraform plan -var="variable_name=value"
```

---

## Example

```bash
terraform plan \
-var="instance_type=t3.micro"
```

---

## Multiple Variables

```bash
terraform apply \
-var="instance_type=t3.micro" \
-var="environment=dev" \
-var="region=ap-south-1"
```

---

## Example Resource

```hcl
resource "aws_instance" "frontend" {
  ami           = var.ami_id
  instance_type = var.instance_type
}
```

Terraform uses the values supplied from the command line.

---

## Advantages

* Fast
* Easy for testing
* Good for temporary overrides

---

## Disadvantages

* Commands become lengthy
* Difficult to manage many variables
* Sensitive values may appear in shell history
* Hard to maintain for teams

---

## Production Usage

Typically used for:

* Quick testing
* Temporary changes
* Emergency overrides

Not commonly used for large-scale deployments.

---

# Method 3: terraform.tfvars File

Terraform automatically loads a file named:

```text
terraform.tfvars
```

without requiring additional arguments.

---

## Example

### terraform.tfvars

```hcl
instance_type = "t3.micro"
environment   = "dev"
region        = "ap-south-1"
```

---

## Command

```bash
terraform plan
```

Terraform automatically reads the file.

---

## Advantages

* Cleaner Terraform commands
* Easy management of variables
* Separates configuration from code

---

## Disadvantages

* A single file becomes difficult for multiple environments
* Easy to accidentally deploy wrong configurations
* Sensitive values may be exposed

---

## Production Usage

Common in:

* Development environments
* Internal testing
* Small Terraform projects

---

# Method 4: Custom tfvars Files

Terraform supports custom variable files.

This is the most common production approach.

---

## Example Files

```text
dev.tfvars
stage.tfvars
prod.tfvars
```

---

## dev.tfvars

```hcl
instance_type = "t3.micro"
environment   = "dev"
region        = "ap-south-1"
```

---

## prod.tfvars

```hcl
instance_type = "t3.large"
environment   = "prod"
region        = "us-east-1"
```

---

## Usage

```bash
terraform plan -var-file="dev.tfvars"
```

or

```bash
terraform apply -var-file="prod.tfvars"
```

---

## Advantages

* Environment separation
* Better maintainability
* Easier collaboration
* Industry-standard approach

---

## Disadvantages

* Additional files to manage
* Sensitive values require protection

---

## Production Usage

Widely used for:

* Development
* Staging
* Production

Example:

```text
environments/
├── dev.tfvars
├── stage.tfvars
└── prod.tfvars
```

---

# Method 5: Auto-Loaded tfvars Files

Terraform automatically loads:

```text
terraform.tfvars
terraform.tfvars.json
*.auto.tfvars
*.auto.tfvars.json
```

---

## Example

### common.auto.tfvars

```hcl
region = "ap-south-1"
```

---

### networking.auto.tfvars

```hcl
vpc_cidr = "10.0.0.0/16"
```

Terraform automatically reads both files.

---

## Benefits

* No need for -var-file
* Useful for shared configuration
* Keeps configurations modular

---

## Production Usage

Often used for:

* Shared settings
* Organization standards
* Global variables

---

# Method 6: Environment Variables

Terraform automatically loads environment variables prefixed with:

```text
TF_VAR_
```

---

## Syntax

```bash
export TF_VAR_variable_name="value"
```

---

## Example

```bash
export TF_VAR_instance_type="t3.micro"
export TF_VAR_environment="dev"
```

---

## Command

```bash
terraform apply
```

Terraform automatically consumes these values.

---

## Mapping

```text
TF_VAR_instance_type → instance_type

TF_VAR_environment → environment
```

---

## Advantages

* Excellent for automation
* No hardcoded values
* Easy secret injection
* Works well with CI/CD systems

---

## Disadvantages

* Harder to debug
* Values are not visible in Terraform files
* Environment drift can occur

---

## Production Usage

Extensively used in:

* GitHub Actions
* GitLab CI
* Jenkins
* Azure DevOps
* Terraform Cloud

---

## GitHub Actions Example

```yaml
env:
  TF_VAR_environment: prod
  TF_VAR_instance_type: t3.large
```

Terraform automatically receives these values.

---

# Method 7: Default Values

Variables can define default values.

---

## Example

```hcl
variable "instance_type" {
  type    = string
  default = "t3.micro"
}
```

---

## Behavior

If no value is supplied from any source:

```bash
terraform plan
```

Terraform uses:

```text
t3.micro
```

automatically.

---

## Advantages

* Simplifies deployment
* Reduces required inputs
* Good for common settings

---

## Disadvantages

* Can hide mistakes
* May lead to unintended deployments
* Not suitable for secrets

---

## Production Usage

Good candidates:

* Regions
* Tags
* Naming conventions
* Feature flags

Avoid for:

* Passwords
* Tokens
* API Keys
* Database credentials

---

# Variable Precedence Order

When the same variable exists in multiple locations, Terraform follows precedence rules.

Higher priority overrides lower priority.

| Priority | Source                            |
| -------- | --------------------------------- |
| Highest  | CLI `-var`                        |
| 2        | CLI `-var-file`                   |
| 3        | `.auto.tfvars`                    |
| 4        | `terraform.tfvars`                |
| 5        | Environment Variables (`TF_VAR_`) |
| Lowest   | Default Values                    |

---

# Variable Precedence Example

## Variable

```hcl
variable "environment" {
  default = "dev"
}
```

---

## terraform.tfvars

```hcl
environment = "stage"
```

---

## Command

```bash
terraform plan -var="environment=prod"
```

---

## Result

```text
environment = prod
```

Reason:

CLI variables have higher precedence than tfvars files.

# Common Interview Questions

## How many ways can variables be passed to Terraform?

Common methods:

1. Interactive Prompt
2. CLI -var
3. terraform.tfvars
4. Custom -var-file
5. *.auto.tfvars
6. Environment Variables (TF_VAR_)
7. Default Values

---

## Which method is most common in production?

Typically:

```text
-var-file
```

for environment configuration.

---

## Which method is most common in CI/CD?

```text
TF_VAR_ environment variables
```

---

## Why avoid interactive prompts in production?

Because:

* deployments should be automated
* manual input causes inconsistency
* CI/CD pipelines cannot provide interactive responses

---

# Production Reality

Enterprise Terraform deployments often involve:

* hundreds of variables
* multiple environments
* reusable modules
* automated pipelines

A well-designed variable strategy improves:

* scalability
* maintainability
* security
* deployment consistency

A poor variable strategy leads to:

* infrastructure drift
* deployment failures
* environment inconsistencies
* operational overhead
