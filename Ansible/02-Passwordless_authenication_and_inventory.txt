PASSWORDLESS AUTHENTICATION & INVENTORY IN ANSIBLE

Passwordless Authentication (Ansible Context)

Ansible primarily uses SSH for communication with managed nodes.

Passwordless authentication means Ansible connects to remote servers using SSH key-based authentication instead of passwords.

Benefits:

* Secure access
* No password prompts during execution
* Required for automation at scale

Flow:

1. Control node initiates SSH connection.
2. Managed node verifies SSH key.
3. Connection established without password.

## AWS EC2 Authentication

In AWS, EC2 instances commonly use Key Pairs for SSH authentication.

During instance creation:

* AWS injects the public key into the EC2 instance.
* User downloads the private key (.pem file).

Because AWS already places the public key inside the instance, manual ssh-copy-id is usually unnecessary.

Important:

* .pem file acts as private key
* Keep it secure
* Used for SSH login and Ansible access

Example SSH connection:

ssh -i ansible_master.pem ubuntu@<PUBLIC_IP>

Where:

* ansible_master.pem -> Private key file
* ubuntu -> Default username for Ubuntu AMI
* <PUBLIC_IP> -> Public IPv4 of EC2 instance

Common usernames:

* Ubuntu AMI -> ubuntu
* Amazon Linux -> ec2-user
* CentOS -> centos

## SSH Key Setup (Non-AWS / Traditional Servers)

1. Generate SSH key pair

Command:
ssh-keygen

This generates:

* Private key -> ~/.ssh/id_rsa
* Public key -> ~/.ssh/id_rsa.pub

2. Copy public key to remote server

Command:
ssh-copy-id user@<SERVER_IP>

Example:
ssh-copy-id ubuntu@192.168.1.10

3. Verify passwordless login

Command:
ssh ubuntu@192.168.1.10

## Inventory in Ansible

Inventory defines:

* Managed nodes (hosts)
* Host groups
* Connection details

It acts as the source of truth for target infrastructure.

Inventory contains:

* Hostnames / IPs
* Groups
* Variables
* SSH connection details

Common connection variables:

* ansible_host
* ansible_user
* ansible_ssh_private_key_file

Default inventory location:
/etc/ansible/hosts

You can also use custom inventory files.

## Static Inventory

Static inventory is manually maintained.

Supported formats:

* INI
* YAML

INI Format Example:

[webservers]
web1.example.com
web2.example.com

[dbservers]
db1.example.com
db2.example.com

[all:vars]
ansible_user=admin
ansible_ssh_private_key_file=/path/to/key.pem

YAML Format Example:

all:
vars:
ansible_user: admin
ansible_ssh_private_key_file: /path/to/key.pem

children:
webservers:
hosts:
web1.example.com:
web2.example.com:

```
dbservers:
  hosts:
    db1.example.com:
    db2.example.com:
```

## Dynamic Inventory

Dynamic inventory automatically fetches infrastructure details from cloud providers or external sources.

Useful for:

* AWS
* Azure
* GCP
* Kubernetes
* Auto-scaling environments

Best for dynamic infrastructures where servers are frequently created or terminated.

Example:

* New EC2 instance launches
* Dynamic inventory automatically detects it
* Ansible can immediately manage it

Common production approach:

* AWS EC2 Inventory Plugin
* Azure Inventory Plugin
* GCP Inventory Plugin

Example:

plugin: amazon.aws.aws_ec2
regions:

* ap-south-1
  filters:
  instance-state-name: running

## Common Ansible Commands

Check connectivity:

ansible -i inventory.ini all -m ping

Run ad-hoc command:

ansible -i inventory.ini all -m shell -a "apt update"

Run playbook:

ansible-playbook -i inventory.ini playbook.yml

## Host Key Checking

When connecting to a host for the first time, SSH asks for fingerprint verification.

Example prompt:

The authenticity of host 'x.x.x.x' can't be established.
Are you sure you want to continue connecting (yes/no)?

Ways to disable this in Ansible:

1. ansible.cfg

[defaults]
host_key_checking = False

2. Environment Variable

export ANSIBLE_HOST_KEY_CHECKING=False

3. Inline Command

ANSIBLE_HOST_KEY_CHECKING=False ansible all -m ping

Note:
Disabling host key checking is useful for labs and temporary environments, but not recommended for highly secure production systems.
